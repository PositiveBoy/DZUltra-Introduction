# DZUltra 用户动线设计文档

> 本文档用于拆解用户从入口到生态落地的完整交互路径，明确每一步的页面形态、交互方式与状态变化。每个步骤均预留「待完善（需要在写代码时进行确认）」区块，供后续细化视觉、动效与具体文案。此文档描绘的是我们最终希望的整个软件实现的状态，前后端与Agent编排都需要参考此文档
>
> 使用方式：这是完整愿景参考，不是每个开发任务都要默认读完。后续 Agent 默认先读 `docs/internal/TECH_AND_INTERACTION_PLAN.md`，再按前端、后端或 Agent 任务读取对应精简规范；只有需要补充具体交互细节时再回到本文档。

## Step 1：入口触发

### 页面形态

- **位置**：大众点评首屏专属入口（如底部 Tab 或顶部金刚区）/ 搜索框一键进入
- **视觉**：入口图标需与主站风格一致
- 具体设计与用户进行确定，其他部分全mock或软件截图图片仿真，只有对应小banner和图标可点击进入

### 交互形态

| 触发方式     | 交互细节                    | 状态变化        |
| -------- | ----------------------- | ----------- |
| 点击专属入口   | 点击进入，可能伴随缩放/渐变转场        | 首页 → 全屏对话页  |
| 搜索框输入关键词 | 识别关键词（如「帮我规划」「行程」）后一键跳转 | 搜索页 → 全屏对话页 |

## Step 2：全屏启动（对话首页）

### 页面形态

- **背景**：浅色底全屏对话窗口，视觉干净、克制
- **顶栏**：左侧返回按钮，中间 AI 图标 + 名称「点仔 Ultra」，右侧「…」菜单按钮
- **中部**：纵向居中布局，从上到下依次为：
  1. 欢迎引导大标题（如「给你安排去哪玩」）
  2. 荧光橙底色记号笔风格标注的重点信息提示（如「**说说 想去哪儿** / **几个人** / **想怎么玩/玩多久 我来规划 ！**」）
  3. 淡蓝色 preset prompt 卡片若干张（横向或纵向排列，数量由实际剩余空间决定）
- **底部**：输入栏（键盘上方），默认显示文字输入框 + 语音按钮 icon，placeholder 文案「发消息或按住说话」
- **模式开关**：输入栏附近放一个轻量开关，文案建议为「路线规划」。默认打开；它只影响模糊输入倾向，不替代自动分流。

### 顶栏交互

| 元素            | 交互细节                            | 状态变化               |
| ------------- | ------------------------------- | ------------------ |
| 左侧返回按钮        | 点击返回大众点评首页或上一级页面                | 对话页 → 上一级（点击时颜色变浅） |
| 中间 AI 图标 + 名称 | 纯展示，不可交互                        | 无                  |
| 右侧「…」菜单按钮     | 点击跳转至 **设置与偏好管理页**（见文末「补充页面 1」） | 对话页 → 设置页（点击时颜色变浅） |

### 底部输入栏交互

| 状态            | 交互细节                                                          | 状态变化             |
| ------------- | ------------------------------------------------------------- | ---------------- |
| **文字输入态**（默认） | 输入框默认聚焦，键盘自动唤起；框内显示 placeholder「发消息或按住说话」；右侧语音 icon 按钮可见      | 空态 → 输入中         |
| **点击语音 icon** | 底部输入栏整体切换为语音模式：框内文字变为「按住说话」，语音 icon 变为键盘 icon（点击可切回文字模式），键盘收起 | 文字模式 → 语音模式      |
| **路线规划开关** | 默认打开，系统偏向生成完整路线；关闭后模糊输入更偏向 answer + related_pois + trace。明显的路线规划、方案微调或新任务仍由系统自动识别 | 路线规划 ↔ 普通问答 |
| **按住输入栏区域说话** | 用户在输入栏任意位置按住，框四周出现蓝色辉光/描边反馈，会随着语音输入而波动外扩；松手后发送语音              | 待命态 → 录制中 → 发送   |
| **上滑取消**      | 录制过程中手指上滑超过阈值，操作区域颜色变化提示「取消」；松手后语音不发送                         | 录制中 → 取消 → 恢复待命态 |
| **切回文字模式**    | 点击语音模式下的键盘 icon，输入栏切换回文字输入态，键盘重新唤起                            | 语音模式 → 文字模式      |

### 中部内容区交互

| 元素           | 交互细节                                    | 状态变化                  |
| ------------ | --------------------------------------- | --------------------- |
| 欢迎引导标题       | 进入时淡入显示，键盘唤起后随页面整体上移；**不会渐隐消失**         | 进入 → 显示               |
| 荧光橙提示文字      | 始终显示在标题下方，作为输入指引                        | 进入 → 显示               |
| 预设 Prompt 卡片 | 淡蓝色底色卡片，点击后自动发送；用户开始输入后若空间不足，卡片可向上缩略或收起 | 默认显示 → 点击发送 / 用户输入后收起 |

### 预设 Prompt 卡片内容策略

预设 prompt 按优先级分为三类，前端按策略展示：

1. **个性化 Prompt**（最高优先级）：若系统持有该用户的持久化偏好信息（常去区域、口味偏好、消费水平等），可生成针对性 prompt
   - 示例：「上次去的川菜馆附近还有啥好吃的？」
   - 示例：「周末想再去徐汇区转转吗？」
2. **热门 Prompt**（兜底优先级）：无个性化数据时展示，由运营/数据端提供当前热门意图
   - 示例：「帮我规划一条周末一日游」
   - 示例：「附近有什么适合约会的餐厅」
   - 示例：「带娃去哪玩一整天」
   - 示例：「深夜还能吃什么」
3. **场景 Prompt**：结合当前时间/季节/天气自动推荐
   - 示例（雨天）：「下雨天的室内好去处」
   - 示例（晚间）：「今晚找个氛围好的酒吧」

### 边界与异常

- 键盘唤起时，中部内容区自动适配，避免被键盘遮挡
- 用户点击 preset 卡片发送后，进入 Step 3 需求输入流程
- 语音无权限时，点击语音 icon 弹出系统权限引导
- 个性化数据为空时，退化到热门 prompt + 场景 prompt

## Step 3：需求输入与 Agent 执行流

> **核心交互理念**：非气泡式对话布局。用户输入置于对话区最上方，Agent 回复与思考链在下方线性展开，整个过程强调「清晰可追踪的执行流」而非即时通讯对话感。大量使用模糊（Blur）效果营造科技简约感。

### 页面形态

- **布局**：全屏对话区域，上下边缘内容渐进模糊消失（`mask-image: linear-gradient`）。无气泡包裹任何内容。
- **内容区结构**（从上到下）：
  1. 用户输入文字（黑色、较大字号、左对齐，固定在对话区顶部）
  2. Agent 思考链（浅色纵向链条，每条链节点对应一个 Agent）
  3. 居中大动画（当前正在执行的步骤）
  4. Agent 阶段性回复/提问（纯文本或卡片，左对齐，无气泡）
  5. 底部输入栏（持续存在，与 Step 2 相同的双模式输入栏）

### 发送后的页面过渡

用户点击发送（文字或语音）后：

| 阶段     | 交互细节                                                                                      | 状态变化            |
| ------ | ----------------------------------------------------------------------------------------- | --------------- |
| 发送瞬间   | 原 Step 2 页面的欢迎标题、荧光橙提示、preset 卡片整体模糊变淡并消失（`filter: blur() + opacity → 0`），过渡时长约 300-400ms | 引导页 → 对话页       |
| 用户输入上屏 | 用户输入的文字以较大字号（约 18-20px）、黑色、左对齐，从底部输入栏位置动画上移到对话区最顶部                                        | 输入栏文字 → 对话区顶部固定 |
| 对话区准备  | 用户输入下方留白，等待 Agent 链开始渲染                                                                   | 空 → 等待中         |

### 用户编辑已发送内容

| 触发             | 交互细节                                                                                                                    | 状态变化       |
| -------------- | ----------------------------------------------------------------------------------------------------------------------- | ---------- |
| 点击对话区顶部的用户输入文字 | 对话部分除该文字外全部模糊（`backdrop-filter: blur()`），文字字号增大（便于编辑），系统键盘弹出，**所有 Agent 执行流程暂停**（已发起的 LLM 请求不会被取消/中断，继续等待响应，但前端暂不处理新结果） | 正常态 → 编辑态  |
| 编辑后重新提交        | 流程完全重新开始，原 Agent 链清空，新请求发送                                                                                              | 编辑态 → 重新发送 |
| 点击文字外的模糊区域     | 取消编辑，文字恢复原字号，对话区变清晰，Agent 流程继续                                                                                          | 编辑态 → 正常态  |

### Agent 思考链展示

> 参考 `/apps/api/app/agents/strategy.py` 中的 `MAIN_PLANNING_AGENT_STRATEGY` 列表定义 Agent 顺序与职责。前端按该顺序渲染链节点。**后续 strategy.py 更新时需同步更新此处渲染逻辑。**

Agent 链以浅色纵向线条连接，每个节点显示：

| 节点元素     | 说明                                                                                         |
| -------- | ------------------------------------------------------------------------------------------ |
| Agent 名称 | 浅灰色细字（如 `ConstraintDiscoveryAgent → 解析需求`）                                                              |
| 状态图标     | 等待中（灰色圆点）→ 执行中（居中大动画）→ 完成（绿色勾）                                                             |
| 执行动画     | 每个 Agent 执行时，屏幕**垂直居中**位置显示一个较大的动画/插图，表示当前正在做的事（如「正在解析你的需求…」「正在搜索附近的餐厅…」）。动画完成后收起，链节点变为绿色勾 |

**Agent 链节点顺序**（与 `AGENT_STRATEGY.md` 和 strategy.py 对齐）：

```
InteractionRouterAgent（判断任务类型：新规划、补全、微调、换方向、普通问答）
    ↓
ConstraintDiscoveryAgent（拆解本轮目标、硬约束、软约束和缺失字段）
    ↓
UserPreferenceAgent（读取历史收藏、评分、去过的店和 UGC 评价偏好）
    ↓
ContextGroundingAgent（调用 POI、UGC、地图、排队、天气、交通等 provider 落地事实）
    ↓
PlanSolverAgent（生成多套可执行候选 plan）
    ↓
PlanEvaluatorAgent（校验硬约束、计算分数拆解、排序）
    ↓
PlanExplanationAgent（生成用户可读解释、风险提醒和推荐菜展示）
```

InteractionRouterAgent 会同时参考用户输入、路线规划开关、当前页面和当前任务上下文。页面状态是强提示，不是硬限制：方案页里“第二个换成川菜”应理解为微调；方案页里“算了，明天带娃去上海迪士尼附近玩”应理解为切换任务或新规划。

> \[!NOTE] (AI 更新占位)
> 以上节点顺序与职责描述依据 `docs/internal/AGENT_STRATEGY.md` 和 `apps/api/app/agents/strategy.py` 编写。
> **当 strategy.py 中的 Agent 列表、职责、顺序发生变化时，需同步更新此处。**
> 更新内容应包含：节点名称、职责简述文案、节点间的跳转条件。

### 自动滚动行为

见全局「内容块导航系统」章节。Agent 链本身不按每个 Agent 节点拆页；一组 Agent 执行状态在同一个内容块内线性播放。用户输入、Agent 执行、补全问题、补全后的 Agent 反应页、需求总结和最终方案页才是主要纵向内容块。新内容块插入时自动 snap 至屏幕顶部，右侧节点指示器同步更新。

- 上下边缘内容使用 CSS `mask-image` 实现渐进模糊消失效果，营造沉浸感

### 用户偏好检测（后台异步）

> 每次用户输入（文字/语音）、补全选择和微调行为，都可以异步交给 UserPreferenceAgent 分析。

| 维度       | 说明                                                           |
| -------- | ------------------------------------------------------------ |
| **检测目标** | 从用户话语中提取**长期偏好**（口味、消费习惯、出行风格等），排除临时意图（如「今天下午想去xx」中的具体地点/时间） |
| **执行方式** | 后台异步运行，用户端不显示等待状态，不阻塞主流程；右侧 Debug Trace 可以展示偏好检测事件 |
| **存储**   | 偏好提取后写入用户档案数据库，标记来源（哪次对话、哪条输入）                               |
| **查看**   | 用户可在「设置与偏好管理页」（补充页面 1）查看已记录的偏好条目，支持手动删除/修正                   |
| **首次使用** | 新用户无偏好档案，仅依赖本轮对话内的临时偏好                                       |

> \[!NOTE] (V2 实现边界)
> V2 先使用本地 JSON Mock 档案模拟“用户档案数据库”，默认文件为 `data/mock/user_preferences.json`。这可以支撑演示中的偏好检测、查看、修正和删除；真实账号体系、线上数据库、LLM 偏好抽取和跨设备同步放到 V3。

**提取示例**：

| 用户输入                | 临时 Intent（不记录） | 提取的 Preference（记录） |
| ------------------- | -------------- | ------------------ |
| 「周末想去徐汇区逛逛，最好有咖啡馆」  | 周末、徐汇区         | 偏好：咖啡馆、步行逛街        |
| 「上次那个川菜馆太爽了，再来个类似的」 | 上次、类似的         | 偏好：川菜、重口味          |
| 「带娃就别太累了，找个公园就好」    | 公园             | 偏好：亲子友好、轻松出行       |

### 边界与异常

- Agent 执行超时：链节点显示橙色警告图标，提示「这一步花了比较久，正在重试…」
- Agent 执行失败：链节点显示红色叉号，下方提示失败原因，用户可点击「重试此步骤」
- 用户编辑重发时：所有已完成/进行中的 Agent 状态清空，链重新开始
- 语音输入同 Step 2 底部输入栏交互逻辑

## Step 4：多轮确认与信息补全

> 在 Agent 执行过程中，若发现关键信息缺失或需要用户偏好澄清，系统会以**新内容块的形式**插入选择卡片到对话区。V2 最多追问 2 轮；第二轮最多 3 个偏好问题。卡片均为内联内容块（非浮层遮罩），与其他内容块一样参与纵向滚动与节点指示器导航。

### 页面形态

- **布局**：卡片作为独立内容块插入对话区，出现在当前 Agent 链节点下方，与其他内容块统一参与内容块导航系统的纵向滚动。无模糊背景遮罩，卡片即为页面滚动流的一部分。
- **视觉**：卡片为浅色底、圆角、与屏幕等宽或略窄；选项以标签/按钮形式排列；底部有确认按钮
- 底部输入栏持续存在，placeholder 根据场景动态变化

### 4.1 关键信息补全卡片

当用户初始输入缺少生成方案所需的关键信息时（由 ConstraintDiscoveryAgent 检测），以新内容块形式插入补全卡片。

**触发条件**：ConstraintDiscoveryAgent / ConstraintDiscoveryAgent 判断用户输入中缺失以下任一阻塞字段：

- 目标城市/区域（无默认值，需要明确或使用当前定位）
- 人数（默认 1 人）
- 时间窗（默认「随便」，但需要用户确认或选择使用系统默认）
- 计划中是否包含饮食（默认「随便」）

**卡片内容**：

| 字段   | 控件类型                                 | 默认值  | 说明                                                                           |
| ---- | ------------------------------------ | ---- | ---------------------------------------------------------------------------- |
| 目标城市/区域 | 自由输入 + 定位/当前城市快捷选项 | 当前城市 | 例如「北京望京」「上海徐汇」「附近」 |
| 人数 | 人数选择器（上下滑动滚轮，类似 iOS 闹钟；也可点击弹出键盘直接输入） | 1 人 | 范围 1-20 人，选择题带「其他」 |
| 时间窗 | 快捷选项 + 时间选择器 | 「随便」 | 快捷选项：「上午」「下午」「晚上」「随便」「其他」；若选具体时段，下方展开开始时间 + 结束时间 |
| 包含饮食 | 多选标签 | 「随便」 | 选项：「随便」「吃主食」「吃小吃」「吃当地特色小吃」「喝饮品」「不吃任何东西」「其他」；「随便」与具体选项互斥 |

- 卡片底部：大号「**确定**」按钮（主色填充）
- 底部输入栏 placeholder：「输入回答或长按语音回答」
- **语音/文字补充**：用户可在输入栏直接说「三个人，下午，吃小吃」，AI 解析后自动填入卡片对应字段，用户确认无误后点「确定」

### 4.2 Agent 自主提问

在 Agent 汇总信息时（天气、定位、附近 POI、User Preference 等），若发现需要用户偏好澄清的场景，Agent 自主生成选择题卡片提问。第二轮最多出现 3 个题；预算、口味、体力、路线风格等都不阻塞规划。

**提问触发示例**：

| 场景                      | Agent 判断逻辑                                 | 提问示例                                                                  |
| ----------------------- | ------------------------------------------ | --------------------------------------------------------------------- |
| 首次用户 + 计划含饮食 + 辣味餐厅排名靠前 | UserPreferenceAgent 发现无辣味偏好数据，但候选 POI 中辣味餐厅靠前     | 「你能接受辣的口味吗？」选项：**无辣不欢 / 微辣可以 / 一点辣都不行 / 随便**（可多选）                     |
| 多种类型 POI 靠前             | ContextGroundingAgent 发现商场、公园、名胜古迹等不同类型均有高排名候选 | 「你现在更想去哪种地方？」选项：**逛商场 / 逛公园 / 名胜古迹 / 小众打卡 / 随便逛逛**（可多选）               |
| 预算相关                    | 候选 POI 人均消费差异大                             | 「这次出行预算大概多少？」选项：**人均 ¥50 以内 / ¥50-150 / ¥150-300 / ¥300+ / 不计预算**（单选） |
| 出行体力                    | 路线步行距离较长                                   | 「能接受多走点路吗？」选项：**多走走没关系 / 不想走太多路 / 最好全程能坐车**（单选）                       |

- 每轮提问以卡片形式呈现，选项可多选或单选（由 Agent 决定），选择题都带「其他」
- 每轮卡片底部有「确定」按钮 + 底部输入栏 placeholder：「输入回答或长按语音回答」
- 用户可在输入栏语音/文字回复，AI 解析后自动选中对应选项
- 提问轮数最多 2 轮；第二轮最多 3 个偏好问题。超过上限后使用默认假设继续，并把假设写入需求总结和 Debug Trace。

### 4.3 需求总结确认卡片

所有 Agent 提问完成后，系统生成一份**需求总结卡片**，汇总所有已确认信息。

**卡片内容**：

```
┌─────────────────────────────────┐
│  确认你的需求                     │
│                                 │
│  目的地：上海徐汇区                │
│  时间：今天下午 14:00-18:00       │
│  人数：3 人                      │
│  偏好：川菜、网红打卡、步行友好      │
│  饮食：吃当地特色小吃              │
│  预算：人均 ¥100-200             │
│  口味：微辣可以                   │
│  ...（动态字段，维度由 Agent 决定）  │
│                                 │
│  [ 确定，开始规划 ]               │
│                                 │
│  💡 若不需要此确认页，可在设置中关闭  │
│     （浅色小字提示）               │
└─────────────────────────────────┘
```

- 卡片底部：「确定，开始规划」按钮
- 底部输入栏 placeholder：「输入或长按语音以进行调整」
- 用户可在确认前通过输入栏修改任意字段
- 若用户在设置中关闭了「需求确认」开关（见补充页面 1），此步骤跳过，直接进入方案生成

### 边界与异常

- 用户跳过非阻塞偏好卡片：使用默认值继续
- 用户跳过阻塞字段：继续停留在补全卡片，或使用当前定位/默认假设时必须明确标记
- 用户在第二轮选择「随便」/「跳过」：Agent 不再继续提问，直接进入总结
- 语音解析失败：toast 提示「没听清，请再试一次」，卡片不动
- 补全卡片字段解析不完整（如只识别到人数，没识别到时间）：已识别字段填入，未识别字段保持默认值

## Step 5：全局动态方案

> Agent 执行流完成后，V2 固定生成 3 套方案，保证演示稳定。V3 可扩展为 3-5 套动态方案。每套方案为一个横向卡片内容块，用户左右滑动切换。

### 页面形态

- **整体布局**：方案区为横向滑动容器（`horizontal scroll snap`），V2 同时显示 3 张方案卡片。当前卡片居中占满屏幕 90% 宽度，左右两侧各漏出相邻卡片的边缘（约 20-30px），暗示可滑动切换。
- **卡片结构**（从上到下）：
  1. 方案标题（悬浮/固定于卡片顶部）
  2. 小地图
  3. 交通信息摘要
  4. POI 地点列表
- 底部输入栏持续存在，上方有一个蓝色半透明毛玻璃按钮「**选用此方案**」
- 卡片均为内容块（无浮层），与右侧节点指示器联动

### 5.1 方案卡片横向切换

| 操作 | 交互细节 | 状态变化 |
| ---- | -------- | -------- |
| 左右滑动 | 手指左右滑动卡片区域，横向 snap 切换方案。自带惯性，松手后吸附至最近卡片 | 方案 N → 方案 N±1 |
| 点击两侧露出边缘 | 点击当前卡片左侧或右侧露出的相邻卡片边缘，自动 snap 切换 | 方案 N → 方案 N±1 |

### 5.2 卡片标题

- 标题由大模型（PlanExplanationAgent）生成，体现本方案的差异化特点
- 示例标题：「**美食扫街·舌尖上的徐汇**」「**文艺打卡·出片率爆表**」「**轻松遛娃·不费爸妈**」
- 标题位于卡片最上方，字号较大（20-24px），加粗
- 卡片**未展开**状态下向下滑动时，标题固定在屏幕顶部（`position: sticky`），直到该卡片完全滚出

### 5.3 小地图与交通摘要

**地图区域**（卡片顶部，约占卡片 1/3 高度）：

- 展示方案中所有 POI 的地点标记（编号 ①②③…），以及串连路线（彩色虚线/实线）
- 地图不可交互缩放（缩略图模式），点击 → 跳转至第三方地图应用查看完整路线
- 地图下方显示三种出行方式的估算：

| 显示项 | 数据来源 | 说明 |
| ---- | -------- | ---- |
| 驾车 | ContextGroundingAgent / map provider | 「约 25 分钟 · 预计 ¥15-30」 |
| 打车 | ContextGroundingAgent / map provider | 「约 25 分钟 · 预计 ¥35-55」 |
| 公共交通 | ContextGroundingAgent / map provider | 「约 45 分钟 · ¥4」 |

### 5.4 POI 地点列表

每个 POI 按路线顺序纵向排列，显示以下信息。字段设计参考 [大众点评与美团POI字段全景分析](../大众点评与美团POI字段全景分析.md)：

#### 餐厅类 POI

| 信息 | 来源字段 | 展示方式 |
| ---- | -------- | -------- |
| 序号 + 名称 | `name` | 粗体大字，如「① 蜀大侠火锅」 |
| 上榜标签 | 榜单（必吃榜/黑珍珠/人气榜等） | 名称旁小徽章，如「必吃榜」「黑珍珠一钻」 |
| 人均消费 | `avgprice` | 「人均 ¥128」 |
| 二级分类 | `categories` 中类 | 「川菜·火锅」 |
| 星级评分 | `star` + 口味/环境/服务 | ★4.5 口味4.6 环境4.3 服务4.5 |
| 推荐菜 | `dishs`（Top 3） | 「招牌：毛肚、鹅肠、红糖糍粑」 |
| 距离 | 距上一站的距离 | 「距上站 800m」 |

#### 景点/休闲类 POI

| 信息 | 来源字段 | 展示方式 |
| ---- | -------- | -------- |
| 序号 + 名称 | `name` | 粗体大字，如「② 徐家汇天主教堂」 |
| 上榜标签 | 榜单（必玩榜/必逛榜等） | 名称旁小徽章 |
| 景区级别 | 分类/标签 | 「5A 景区」「历史建筑」 |
| 门票/人均 | `avgprice` | 「门票 ¥20」或「免费」 |
| 开放时间 | `business_hour` | 「今日 09:00-17:00」 |
| 距离 | 距上一站的距离 | 「距上站 1.2km」 |

#### 其他类型 POI（饮品/甜品/购物等）

参照上述两类，展示对应的关键字段。所有 POI 均有：
- `headPic` 缩略图（左侧小方图）
- 美团/大众点评 POI 详情页跳转链接（`mShopInfoUrl` / `appShopInfoUrl`）

### 5.5 卡片全屏展开

点击卡片 → 卡片以动效**全屏展开**，变成独立一页：

| 区域 | 内容 |
| ---- | ---- |
| 顶栏 | 左侧「×」退出按钮，中间方案标题 + 下方方案总时长（如「全程约 4 小时 30 分」），**无**「…」菜单 |
| 导航 | 左右两侧边缘有半透明箭头按钮，点击切换至相邻方案的全屏展开视图 |
| POI 详情 | 每个 POI 展开为更详细的信息块：简单文字介绍（`shopDesc`）、部分 UGC 评论截取（`ugcs` Top 2-3 条）、美大 POI 页跳转链接按钮 |
| 底部 | 输入栏 + 蓝色「选用此方案」按钮持续存在 |

- 全屏展开后，该卡片仍为内容块，无模糊遮罩
- 点击「×」退出全屏 → 卡片缩回原位，回到横向滑动方案列表

### 5.6 边界与异常

- 方案仅 1 套时：不显示两侧边缘，不支持滑动切换
- 方案生成中：卡片区显示 skeleton 占位（三张灰色骨架卡片横向排列）
- 方案生成失败：卡片区展示错误提示 + 「重新生成」按钮
- 地图加载失败：地图区域显示占位图 + 文字列出地址

## Step 6：灵活微调

> 用户在浏览方案卡片时，可直接通过底部输入栏对当前方案进行针对性微调。系统默认将输入理解为「针对当前屏幕展示方案」的修改，无需指定方案编号。

### 6.1 默认语境

- 当前屏幕展示哪个方案卡片，底部输入栏的 placeholder 为「**试试说「换掉咖啡馆」或「不想吃辣」**」
- 用户发送的指令默认作用于当前方案
- 系统解析指令后，仅修改当前方案中被提及的 POI 或属性，其余 POI 保持不变

### 6.2 指令类型与前端行为

| 用户指令示例 | 前端行为 | 说明 |
| ---- | -------- | ---- |
| 「把咖啡店换掉」「不想喝咖啡」 | 替换当前方案中所有咖啡/饮品店为其他类型 POI，方案卡片局部刷新对应 POI 行（带替换动画） | 仅替换，不重新跑全链路 |
| 「第二个换成川菜」 | 替换第 2 站 POI 为川菜类候选 | 单站替换 |
| 「保留第一个和第三个，其他重新推荐」 | 锁定 1、3 站，其余站重新从候选池生成 | 部分保留 + 局部重新生成 |
| 「重新生成一个方案」 | **放弃当前方案**，全链路重新跑，生成新方案替换当前方案 | 完全重置 |
| 「生成新的方案」 | 同上 | 完全重置 |
| 未提及保留/替换/重生的自由输入 | 系统判断意图后归类到上述逻辑之一 | — |

- 修改结果：被替换/更新的 POI 行带短暂高亮闪烁动画（约 1 秒），提示用户变更位置
- 全局更新：方案标题可能随之变化（大模型重新总结）
- 底部 toast：「已更新」/「已替换第 2 站」

### 6.3 「选用此方案」按钮

- 位置：底部输入栏**上方**，蓝色半透明毛玻璃背景（`background: rgba(0,122,255,0.15); backdrop-filter: blur(10px)`），圆角胶囊
- 文案：「**选用此方案**」
- 交互：点击 → 进入 Step 7 生态落地（方案转为所选方案的详细确认页）
- 验证：若方案因网络等原因未完整加载，按钮 disabled，提示「方案加载中…」

### 6.4 边界与异常

- 用户连续微调 5 次以上：弹出轻提示「需要我重新帮你规划吗？」，引导一次性重新生成
- 微调请求超时：toast「修改失败，请重试」，卡片不变化
- 语音指令识别失败：toast「没听清，请再说一次」

## Step 7：生态落地

> 用户点击「选用此方案」后，方案转化为一版**详细确认与执行视图**，包含时间线、交通方式选择、一键操作入口与 To-do 清单。

### 7.1 详细方案页面形态

选中方案后，Page 转换为「方案确认与执行视图」（仍为内容块，右侧节点指示器新增一个内容块节点）：

| 区域 | 内容 |
| ---- | ---- |
| 顶栏 | 左侧返回（回到方案浏览），中间方案标题，无「…」菜单 |
| 时间线概览 | 「**距出发还有 1 小时 20 分**」——倒计时/提醒语，红色醒目；下方为该方案的总时间 + 总费用估算 |
| 交通选择器 | 分段控制器：「驾车」「打车」「公共交通」三个 tab，默认选中推荐方式。切换后方案内的交通时间/费用同步更新 |
| POI 时间线 | 每个 POI 节点（纵向时间线左侧圆点 + 时间标签），包含以下操作入口 |

### 7.2 POI 节点信息与操作入口

每个 POI 节点展开显示：

| 内容 | 说明 |
| ---- | ---- |
| 时间标签 | 预计到达时间 / 建议停留时长 / 离开时间 |
| POI 基本信息 | 名称、地址、评分、距离上一站 |
| **一键导航** | 按钮，点击唤起地图应用（传入该 POI 坐标） |
| 排队按钮 | 若 `queueable = true` 且当前有排队提示 → 显示「**在线排号**」按钮，点击进入美大排号页 |
| 团购/优惠券按钮 | 若 `dealInfo` 非空 → 显示「**团购券 ¥XX**」按钮，点击进入美大团购页 |
| 买票按钮 | 若 POI 为景点且需购票 → 显示「**在线购票 ¥XX**」按钮，点击进入美大购票页 |
| 预订按钮 | 若 `bookable = true` → 显示「**预订**」按钮 |

每个 POI 节点可折叠/展开，默认第一站展开，其余折叠。

### 7.3 分享

- 方案顶部/底部有「**分享**」按钮
- 点击 → 调用美团 AWP（App Web Platform）生成网页链接（本产品做一个mock接口就行）
- 链接包含方案的所有 POI、路线、时间信息（以 H5 页面呈现）
- 支持转发至微信、朋友圈、微博等平台
- 生成中：分享按钮显示 loading spinner；生成完成：弹出系统分享面板

### 7.4 大众点评首页同步

选用方案后，**后台自动同步**至大众点评主站：

| 同步内容 | 展示位置 |
| ---- | -------- |
| 「您有一个预定好的计划」 | 大众点评首页——首屏小图标下方、中部区域 |
| 卡片信息 | 方案标题 + 时间 + 第一站 POI 名称 + 当前状态（如「距出发 1h20m」） |
| 点击行为 | 点击卡片 → 回到 DZUltra 详细方案页（Step 7） |

### 7.5 To-do 清单

方案页面最下方，显示一个 **To-do 清单**内容块：

| 事项类型 | 展示内容 | 操作 |
| ---- | -------- | ---- |
| 排队 | 「蜀大侠火锅 · 建议提前排号」 | [在线排号] |
| 买票 | 「徐家汇天主教堂 · 门票 ¥20」 | [在线购票] |
| 预约 | 「XX 餐厅 · 建议提前预订」 | [预订] |
| 团购 | 「XX 咖啡馆 · 团购券 ¥9.9」 | [查看团购] |

- 每项左侧有复选框（点击打勾），标记完成状态
- 已完成的项下沉并变灰，未完成的保持醒目
- 所有链接均跳转至美大对应功能页
- 均完成后，显示在 To-do 首行，文案「已充分准备好，玩得开心！」

### 7.6 边界与异常

- 未登录 → 分享/收藏/排号/购票均需先引导登录
- 地图应用未安装 → 提示「请先安装地图应用」

- 方案已过期（时间已过）→ 方案顶部提示「该方案已过期，是否重新规划？」
- To-do 事项全部完成 → 清单顶部显示「🎉 一切就绪，出发吧」

## 全局交互与状态

### 设计语言：Blur（模糊）体系

本软件充分使用 Blur 效果营造科技简约感，以下为全局模糊使用规范：

| 场景                 | 模糊方式                                                                                                 | 说明                                |
| ------------------ | ---------------------------------------------------------------------------------------------------- | --------------------------------- |
| Step 2 → Step 3 过渡 | `filter: blur()` + `opacity → 0`                                                                     | 欢迎页内容模糊淡出                         |
| 用户点击已发送文字编辑        | `backdrop-filter: blur()`                                                                            | 对话区除用户输入外全部背景模糊                   |
| 上下边缘内容溢出           | CSS `mask-image: linear-gradient(to bottom, transparent 0%, black 10%, black 90%, transparent 100%)` | 对话滚动区上下边缘渐进模糊消失                   |
| 用户输入胶囊冻结           | `backdrop-filter: blur()` + 半透明背景                                                                    | 向下滚动后用户 prompt 冻结为顶部毛玻璃胶囊         |
| 胶囊展开面板             | `backdrop-filter: blur()`                                                                            | 点击胶囊展开后其余页面内容全模糊，面板内显示所有用户 prompt |

### 用户 Prompt 冻结胶囊

大模型在用户首次发送 prompt 后，对该请求进行总结提炼，生成一句简短摘要（如原始输入「周末下午想在望京附近转转，找个咖啡馆坐坐然后吃点好的」→ 总结为「**下午在望京附近玩**」）。该摘要替代原始 prompt 作为胶囊显示文本。

当用户在对话区向下滚动超过用户输入文字区域后：

| 阶段        | 交互细节                                                                                                                                                      |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 触发        | 用户输入文字区域滚出视口                                                                                                                                              |
| 冻结        | 将大模型总结出的摘要以透明模糊毛玻璃（`backdrop-filter: blur()`）的圆角胶囊呈现，水平居中锁定在屏幕顶部，单行显示，溢出部分用 `...` 截断                                                                      |
| 点击胶囊→展开   | 胶囊展开为全屏面板（其余页面内容模糊 `backdrop-filter: blur()`），显示**本轮 Plan 下用户的所有输入 prompt**（原始文字，按时间/节点排序，可有多条：首次 prompt + 后续补充 + 修改重发等），每条可点击 → 自动滚动跳转至该 prompt 对应的内容块节点 |
| 点击展开面板外区域 | 面板收起，胶囊恢复，对话区恢复清晰                                                                                                                                         |
| 返回        | 用户向上滚动至顶部 → 胶囊自动消失，文字恢复原位                                                                                                                                 |

### 内容块导航系统

整个交互系统的纵向滑动以**内容块**为单位组织。这里的“每一页”不是传统路由页面，而是一个可吸附的滚动锚点；一段对话从用户输入到最终方案始终保留在同一条纵向流里，用户可以从头滑到尾，也可以从当前结果向上回看任意阶段。内容块之间无显性边框分隔，仅靠内容间距与滚动节奏区分。

实现时要避免把内容块做成互斥页面切换：用户输入块、Agent 链块、补全块、总结块、方案块应按时间顺序追加到同一个 MobileShell 内部滚动容器里。状态机只负责决定哪些块已经可见，以及当前应自动吸附到哪个块；它不负责卸载或覆盖前面已经出现的块。

**内容块拆分口径**：

- 用户首次输入算 1 个内容块，锚点落在用户需求文字处。
- Agent 执行流算 1 个内容块，Agent 链节点在块内依次点亮；每个 Agent 节点不单独算页。
- 关键信息补全卡片每轮算 1 个内容块。
- 用户完成补全后，需要插入 2-3 个 Agent 反应/过渡思考内容块，例如“已吸收补全信息”“正在重新检索候选地点”“正在组合可解释路线”。这些块帮助用户理解系统不是突然跳到总结或方案。
- 需求总结确认卡片算 1 个内容块。
- 最终 3 套方案整体算 1 个纵向内容块，块内使用横向 snap 切换方案；右侧竖向指示器不为 3 套方案分别增加节点。
- 选用方案后的执行视图和 To-do 清单可各算 1 个内容块。

**右侧节点指示器**：

- 屏幕右侧垂直居中位置显示一列点状节点（每个点对应一个内容块），纵向排列
- 当前所在内容块的节点高亮（实心/放大/主色），已浏览过的节点为浅色，未到达节点为灰色
- 用户可**按住并上下滑动**该指示器实现快速跳转（类似进度条/章节选择器），松手后页面平滑滚动至对应内容块
- 指示器在页面静止 2 秒后自动半透明淡化，触摸屏幕时恢复

**滚动行为**：

- 纵向滑动基于速度与惯性，使用**仿苹果重力滑动**（`-webkit-overflow-scrolling: touch` + 自定义 momentum 曲线），达到极其丝滑的自然减速手感
- 每滑动到一个新的内容块，该块自动吸附至屏幕最上方（snap-to-top，非强制 snap，仅在松手速度低于阈值时触发）。自动吸附只改变滚动位置，不清空或替换前面的对话内容
- 松手时若速度足够快则跨越多个内容块，自然减速停止后吸附至最近的内容块顶部
- 新内容块出现时，系统自动滚动到该块的关键文字处：用户输入文字、问题标题、过渡思考标题、需求总结标题或“给你 3 套方案”

**内容块内部滚动**：

- 单个内容块（如补全卡片、需求总结卡片、方案卡片）内部可以继续上下滑动，承载较长字段、多个 POI 或较长说明
- 当用户在一个内容块内部滑完内容后，再继续向下滑，才进入并吸附到下一个内容块顶部
- 内容块内的子滚动不影响右侧节点指示器，指示器仅在跨越内容块边界时切换

### 网络状态

- 加载中：skeleton / loading spinner
- 无网络：空态页 + 重试按钮
- 弱网：降级展示，优先文字后图片

### 错误处理

- 大模型超时：提示「思考中…」或「稍后再试」
- 语音识别失败：提示「没听清，请再说一次」
- 定位失败：提示手动选择城市

### 手势与动效

- 全局支持下拉刷新（方案页）
- 页面转场统一使用滑动或淡入淡出
- 关键操作（收藏、发送）需有触觉反馈（Haptic Feedback）

    │ 判断 new_planning_task / chat_answer / answer_clarification / refine_current_plan / switch_task
