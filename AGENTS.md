# AGENTS.md

本文件是 DZUltra 项目的顶层协作规则。所有在本项目中工作的 AI Agent 都要先遵守这里的说明，再根据具体任务阅读相关代码和文档。

## 项目名称

DZUltra

## 项目简介

“点仔 Ultra”是面向大众点评“点仔”的 AI 本地路线智能规划升级方案。更完整的项目介绍写在 `README.md`，这里不重复展开。

当前项目已进入 V3 最终接入阶段。所有 Agent 先记住 V3 的核心口径：

- V3：主链路优先使用真实 provider，也就是 LongCat LLM、高德地图 Web 服务 API、彩云天气 API。
- V3：排队、UGC 摘要、推荐菜、用户历史行为等大众点评深度字段继续使用本地 Mock 或平台内 Mock 生成器，但这些数据不能在首屏默认展示给用户。
- V3：真实 provider 超时、失败、返回结构不合法或缺少 Key 时，才允许 fallback 到 Mock；fallback 必须写进 Debug Trace，说明触发原因、使用了哪些 Mock 数据、哪些字段可靠性是 `mocked`。
- Web 端首次进入必须是干净状态：Debug Trace、History、Mock 数据面板都不预置静态 run；用户交互后才生成当前 run。
- 推荐结果必须能解释，Debug 端必须能看懂 Agent 全链路。

## 沟通规则

- 每次回答都要用项目新人也能听懂的话解释。不要默认用户熟悉大型项目、工程术语或 Agent 架构。
- 如果必须使用技术词，要顺手解释它的意思。例如：“Mock 数据，也就是先用假的样例数据模拟真实数据”。
- 先说结论，再说原因。复杂任务要拆成清楚的小步骤。
- 不要只给抽象建议。能落到具体文件、具体命令、具体改动时，就说清楚。
- 如果不确定，要明确说“不确定”，并说明下一步如何确认。
- Agent 设计、Debug 面板、评分策略、真实地图 API 边界等会影响产品方向的问题，要及时向用户确认。
- 已确认的 V3 口径不要反复询问：路线规划模式默认打开但只是意图倾向、每次输入仍需自动分流、最多追问 2 轮、默认返回 3 个可解释方案且后续可扩展到 3-5 个、普通问答返回 `answer + related_pois + trace`、地图距离由 provider 计算。

## 文件安全规则

- 不要乱删文件。
- 不要在没有明确要求的情况下删除、覆盖、重命名已有文件。
- 修改文件前，先理解这个文件在项目里的作用。
- 如果发现已有改动不是自己做的，默认它是用户或其他协作者的工作，不要回滚。
- 涉及大范围改动、迁移、清理、格式化前，要先说明影响范围。

## 开发原则

- 优先保持 Demo 稳定、可演示、可解释。
- V3 优先接真实 LLM、地图和天气 provider；Mock 只能作为可解释 fallback 或深度本地生活字段的临时数据源。
- V3 的 Agent 流程要保持完整：识别意图、收集上下文、结构化提问、偏好分析、检索候选、地图距离、天气约束、排程、约束检查、生成方案、排序、微调。
- 分流要结合页面状态和任务上下文：当前页是补全、确认、方案、选用还是普通问答，会影响用户输入的解释，但不能硬限制用户切换任务。
- LLM 不应该凭空判断坐标、距离、通勤和营业约束；这些事实必须来自 mock provider、地图 API 或 deterministic tool，也就是可复现的规则工具。
- 路线推荐逻辑要能解释：为什么推荐这些 POI、为什么这样排序、哪些用户偏好影响了结果。
- Mock 数据要尽量贴近真实大众点评使用场景，避免过于理想化；但首屏不主动展示基础 Mock 数据，只有 fallback、Mock 生成器预览或当前 run 用到时才展示。
- 用户端体验要贴近移动端大众点评的直觉：信息清楚、行动路径短、推荐理由容易理解。
- Debug 端要帮助看懂 Agent 全链路：输入、约束、候选 POI、筛选理由、排序过程、最终路线和二次调整过程。

## 代码工作方式

- 开始改代码前，先阅读相关目录和已有实现，不要凭空新建一套风格。
- 优先沿用项目已有的框架、命名、组件、数据结构和目录组织。
- 改动尽量小而清楚，只改完成当前任务真正需要的地方。
- 如果新增逻辑影响路线生成、用户偏好、POI 筛选或展示结果，要考虑如何验证。
- 完成后要说明改了什么、为什么这样改、怎么验证。
- 要用 conda activate agent 环境；自动化命令里优先使用 `conda run -n agent ...`。

## 面向新手的解释标准

回答里尽量做到：

- 用普通话说清楚“这一步在解决什么问题”。
- 把复杂系统拆成“输入、处理、输出”来解释。
- 解释 Agent 时，可以理解为“负责某个子任务的小助手”。
- 解释端到端时，可以理解为“从用户输入到页面展示都能跑通”。
- 解释 Debug 端时，可以理解为“给开发者看的过程回放和检查面板”。

## 项目优先级

当前 V3 阶段优先级从高到低：

1. Agent 流程稳定、可解释，并优先走真实 LLM、地图和天气 provider。
2. 用户端首次进入干净，交互后才展示真实 run；Mock 只在 fallback 或生成器场景出现。
3. Debug Trace 能看懂完整链路，包括真实 provider 调用、候选池、排除理由、约束检查、排序、耗时和 fallback 原因。
4. API、schema 和 provider 边界清楚，方便替换真实 POI/UGC/排队/用户历史数据源。
5. 再考虑性能、扩展性和线上工程化细节。
